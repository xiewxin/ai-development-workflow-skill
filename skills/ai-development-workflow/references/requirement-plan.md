# 需求計畫指南

使用本指南時，先讀取並明確連結[需求計畫範本](../assets/requirement-plan-template.md)。新文件以該範本為骨架；更新既有文件時，遵循專案既有格式與章節慣例。

## 來源與證據優先級

需求來源預設可為任務系統，也可直接來自原型、知識庫、表格或目前對話。來源衝突時依序採用：

1. 目前對話的明確說明。
2. 需求內容及驗收條件。
3. 原型、知識庫或表格。
4. 已驗證的程式碼、測試與實作文件。
5. 歷史計畫。

同一優先級內仍有衝突時，分別列出事實、影響與待確認問題，不自行選擇。

### 任務系統

閱讀標題、描述、預期結果或驗收條件、圖片、關聯資料、附件、全部留言，以及留言中的連結。只有下列情況才讀變更歷史：

- 目前內容彼此衝突。
- 有跡象顯示需求曾變更。
- 使用者明確要求追溯變更。

來源無法存取、附件不可讀或內容缺失時，在「待確認事項」列出缺口與影響，不猜測內容。

### 原型

原型閱讀範圍包含：

- 使用者提供連結所指向的頁面。
- 從該頁依互動可到達的頁面。
- 與同一流程直接相關的頁面。

若範圍不明或過大，先列出建議範圍與排除範圍，再只問一個最能界定範圍的確認問題。

## 主倉庫探索

預設目前工作目錄是主倉庫。依序閱讀並驗證：

1. 根目錄規範檔與貢獻、開發流程說明。
2. 與需求相關的 `docs/agents/` 指南。
3. 實際程式碼、設定、路由或入口、資料流與測試。
4. 可能被 Git 忽略的 `docs/plans/` 文件。

`docs/plans/` 只代表歷史意圖，不可單獨作為現況證據。`docs/agents/` 也必須由目前程式碼與測試驗證；若內容不一致，在「現況與證據」指出文件漂移及其影響。

每筆證據先標示所屬「專案／倉庫」，再使用該倉庫內的相對路徑；必要時附類別、方法、設定鍵或測試名稱。不要使用本機絕對路徑，也不要貼大段程式碼或日誌。

## 跨專案邊界

只有發現具體跨專案缺口時，才詢問或探索其他專案，例如：

- client 存在但找不到 server。
- producer 存在但找不到 consumer。
- 需求依賴的資料表、Model 或 Repository 缺失。
- 只找到介面呼叫，找不到實作端。
- 已驗證鏈路只覆蓋完整流程的一段。

詢問時必須列出缺口、已查的專案／倉庫、倉庫內相對路徑、搜尋詞與現有證據。若未取得其他專案資訊，仍完成主倉庫可確認的計畫，將跨專案部分標記為待驗證，不以推測補齊。

跨專案預設只在主倉庫建立需求計畫；各專案分別維護自己的實作文件。

## 必填章節

每份計畫都要保留以下章節；無內容時明確填寫「無」並說明判斷依據，不省略章節。

1. **基本資訊／狀態**：主題、日期、負責範圍、目前狀態與前版連結。
2. **需求來源與目標**：來源、問題、目標、驗收條件與來源缺口。
3. **範圍與非範圍**：本次包含、不包含及跨專案邊界。
4. **現況與證據**：目前入口、合同、資料流、測試、文件漂移、所屬專案／倉庫與倉庫內相對路徑。
5. **影響分析**：介面、資料、事件、相依服務、相容性與使用者流程。
6. **複用分析**：現有能力、搜尋結果與複用決策。
7. **方案與決策**：選定方案、替代方案、取捨與不變條件。
8. **風險**：風險、觸發條件、緩解與驗證方式。
9. **檔案級實作計畫**：逐檔案列出所屬專案／倉庫、倉庫內相對路徑、修改目的、符號與預期行為。
10. **實作清單**：可勾選、可驗證且依相依順序排列的工作項目。
11. **測試與驗證**：記錄驗證策略、範圍、風險、所屬專案／倉庫、倉庫內相對執行位置、命令或方式、整體通過標準、回歸邊界與測試設計文件連結。
12. **文件改動**：二選一記錄各專案需更新的實作文件與狀態，或「無需更新」及其依據。
13. **待確認事項**：只保留會影響範圍、合同、方案或驗收的問題。
14. **實作與驗證結果**：逐筆列出所屬專案／倉庫、倉庫內相對執行位置、實際變更、執行指令、結果、未完成項目與剩餘風險。
15. **AI 協作成效**：固定填寫指定的簡短欄位。

## 複用分析

對每個候選能力記錄所屬專案／倉庫、class、method 與倉庫內相對 path，並在「複用、擴充、抽取、不複用」中選一項。說明決策原因、適用邊界與必要調整。

找不到可複用能力時，不寫籠統結論；列出使用過的搜尋詞、搜尋路徑與未命中的結果，再說明為何需要新增能力。

## 分層與方案原則

分層優先遵循專案規範與附近模組。只有專案本身採用 Controller、Logic、Service、Repository，且需求與現況沒有衝突時，才把 `Controller → Logic → Service → Repository` 作為候選鏈路；Repository 原則上只由 Service 呼叫。

其他架構沿用原有分層，不因本指南強迫重構。計畫需說明採用的分層依據，以及新邏輯放置位置的理由。

## 測試、驗證與命令回填

「測試與驗證」及「實作與驗證結果」中的每筆命令，都要標示所屬專案／倉庫與執行位置。執行位置使用倉庫根目錄或倉庫內相對路徑，不使用本機絕對路徑；即使不同倉庫使用相同命令，也要分列記錄。

需求計畫只定義驗證策略、驗證範圍、主要風險、命令或方式、整體通過標準與回歸邊界，並以相對連結指向獨立測試設計文件。詳細測試情境、前置條件、操作步驟與逐案例預期結果只放在獨立 test-design，不在需求計畫重複維護。

## 計畫落盤

先遵循倉庫既有 `docs/plans/` 命名與格式慣例。沒有慣例時，使用：

```text
docs/plans/YYYY-MM-DD-<topic>.md
```

`topic` 使用簡短、穩定的 kebab-case，不納入易變狀態或人名。不可覆蓋歷史計畫；同名時建立 `-v2`、`-v3`，並在新版本引用前一版與取代原因。

## 狀態、核准與回填

狀態只能使用：草擬、已核准、實作中、已完成、部分完成、已阻塞、已取代。

- 草擬：仍待範圍或方案核准。
- 已核准：使用者已明確核准計畫與實作範圍。
- 實作中：正在執行已核准項目。
- 已完成：核准範圍的實作、驗證與文件回填全部完成。
- 部分完成：部分核准項目完成，其餘項目與原因已列明。
- 已阻塞：存在無法在目前授權或資訊下排除的阻塞。
- 已取代：已有引用本文件的新版本。

實作前必須取得明確核准。實作中出現新事實或需偏離計畫時，先列出原方案、新證據、影響與可選方案；確認後更新計畫，再繼續受影響部分。

完成時依專案／倉庫回填實際執行位置、指令、結果摘要、未完成項目與剩餘風險。未完成文件更新或必要驗證時，不得標記為已完成。

## 文件改動

依實作所在專案與既有慣例更新文件。既有模組進行迭代時，同步更新相關實作文件；新模組沒有慣例時，提出建議路徑與必要性，取得確認後再新增。

「文件改動」必須二選一：

- **需要更新**：列出所屬專案／倉庫、倉庫內相對路徑與更新內容，使用 checkbox 追蹤完成狀態。
- **無需更新**：移除更新 checkbox，以普通文字寫「無需更新」及判斷依據，不可形成未完成項。

跨專案時，各專案只維護自己的實作文件，需求計畫仍只放在主倉庫。環境提供 `agents-doc-maintainer` 時，可用它檢查與維護模組文件。

## AI 協作成效

只保留以下簡短欄位，不建立或維護跨需求 CSV：

- 需求規模。
- 使用的需求計畫／測試設計／Git Diff 審查。
- 傳統預估。
- 實際耗時。
- 節省比例。
- 具體發現。

## 閱讀性與安全

- 結論優先，章節與段落保持短小。
- 事實、假設與問題分開陳述。
- 證據只使用相對路徑與必要識別字。
- 不貼大段程式碼、日誌或來源全文。
- 不在不同章節重複相同背景。
- 範例與占位內容必須虛構，不寫入公司、專案、需求識別碼、內部網址、絕對路徑、憑證或個人資料。
