# 測試設計指南

使用本指南時，先讀取並明確連結[測試設計範本](../assets/test-design-template.md)。新文件以該範本為骨架；更新既有文件時，遵循專案既有格式與章節慣例。

## 文件邊界

- 需求計畫中的測試章節只記錄測試策略、範圍、風險與可重現命令。
- 獨立測試設計記錄詳細情境、前置條件、步驟、預期結果、邊界與回歸範圍。
- 測試設計只引用目標測試路徑，不放完整測試程式碼；只有在澄清關鍵斷言或資料流時，才使用極短偽碼或片段。

## 按需求規模裁剪

預設產出能直接執行的最短版本，不以篇幅代表完整性。先覆蓋明確需求與具證據的高風險，再依規模選擇結構：

- **小型**：單一入口、有限合同或少量驗收情境。優先用緊湊表格；四個驗收情境通常應在約 100 行內交付。
- **中型**：涉及多個責任層、合同或測試層級。可依模組拆表，但只拆解確有不同前置條件或驗證方式的部分。
- **大型**：跨專案、多資料流或多階段交付。才按專案、流程或風險分節，並保留跨專案執行位置。

同一事實只寫一次，其他章節以情境 ID 或相對連結引用：

- 「範圍與策略」只界定目標、邊界與方法，不重列逐案例預期。
- 「測試資料策略」是規劃階段資料建立、隔離與清理方式的唯一來源。
- 「詳細情境」是行為、前置條件、步驟、預期結果、優先級與回歸保護的唯一來源。
- 「目標自動化映射」是預計測試檔案與名稱、mock／外部邊界、關鍵斷言、執行位置、預計命令與預期輸出的唯一來源；資料策略只引用情境 ID。
- 「回歸」只記錄跨情境或既有套件的回歸方式，不複製情境內容。
- 「交付測試清單」只記錄情境 ID 與交付狀態摘要，不重述步驟、資料、檔案、命令或結果。
- 「自動化測試實施結果」屬實作後生命週期，可記錄實際檔案／名稱、實際命令／結果與設計偏差；仍以情境 ID 回指原設計。

## 建立時機

- 完整流程中，於需求計畫核准後、實作前建立。
- 使用者只要求測試設計時，可獨立建立，不得立即新增或修改目標專案的實際測試目錄。
- 只有進入已核准的實作階段，才在目標專案實際採用的測試目錄建立或修改測試。`/tests` 只代表該實際目錄，不固定代表 `tests/Unit`。

## 存放與連結

優先遵循目標倉庫的命名與文件慣例。沒有慣例時使用：

```text
docs/plans/YYYY-MM-DD-<topic>-test-design.md
```

- `topic` 使用簡短、穩定的 kebab-case。
- 不覆蓋歷史文件；同名時建立 `-v2`、`-v3`，並說明取代原因。
- 測試設計、需求計畫與需求來源必須互相連結；無法直接連結的來源，記錄可追溯的中立摘要與缺口。
- 跨專案需求預設只在主倉庫維護一份測試設計。每筆目標測試、證據與命令都要標示專案／倉庫，以及倉庫內相對路徑或執行位置。

## 情境設計

先逐項覆蓋需求明確指定的驗收情境，再檢查正常、邊界、無效、異常、權限、相容性與回歸中哪些確實適用。不得為展示完整性主動新增需求未要求的案例；只有目前程式碼證據顯示下列明確風險時才補充，並在情境中用一句話標示證據路徑或 symbol：

- 安全風險。
- 資料正確性風險。
- 主要副作用風險，例如重複寫入、漏寫或錯誤狀態轉換。
- 高回歸風險，例如已驗證的既有呼叫端或關鍵相容合同。

每個適用情境必須包含：

1. 唯一情境 ID。
2. 對應需求、驗收條件或風險。
3. 前置條件。
4. 可重現的操作步驟。
5. 可觀察的預期結果。
6. 驗證類型：自動化、手動、環境或上線觀察。
7. 優先級與排序依據。
8. 需保護的回歸範圍。

優先級先沿用目標專案既有尺度；專案沒有定義時，使用下列測試排序語意：

- **P0**：安全事故、不可逆資料損毀或服務不可用風險，最先驗證並阻斷交付。
- **P1**：核心驗收、主流程或對外合同，交付前必須驗證。
- **P2**：適用的邊界、錯誤、相容性、主要副作用或高回歸風險。
- **P3**：低風險補充或觀察性驗證，不影響已明確要求的驗收範圍。

每個情境都要附排序理由；優先級只決定順序，不得用來刪除需求明確指定的情境。

小型需求預設每個情境一列，只有單列無法清楚表達多步驟、不同環境或多個可觀察結果時才展開子章節。手動、SQL、權限、環境與上線觀察全部不適用時，整個對應章節只寫一行「無。依據：…」。

不得因時限、主管壓力或交付壓力，擅自刪除需求中的無效、過期、未提供或其他驗收條件。若指令與驗收條件衝突，列出受影響情境、回歸風險與可選方案，再請使用者確認。

## 自動化映射

依目標倉庫實際的 test runner、測試目錄與鄰近測試設計，不強迫使用特定框架或 `tests/Unit`。

每個自動化情境都要在一張緊湊表格映射：

- 專案／倉庫與預計採用的測試檔案相對路徑。
- 測試名稱或命名規則。
- mock、外部依賴與時間、亂數或網路邊界。
- 關鍵斷言，不只是方法被呼叫。
- 執行位置、可重現命令與預期輸出。

資料建立與清理只在「測試資料策略」定義，本節以情境 ID 引用。相同檔案、命令或 mock 邊界可合併記錄一次，再由情境 ID 對應；不要為每個案例複製共同設定。

不要求所有情境都自動化。無法自動化時，說明技術、成本、環境或觀測性限制，並指定手動、環境或上線觀察的替代驗證。

## 測試資料

優先級由高到低：

1. mock、factory、fixture、測試專用 seed 或測試自行建立的可控資料。
2. 由 transaction rollback 隔離與回復的測試資料庫資料。
3. 受控、可版本化且可重建的環境 seed。
4. 不得已才使用既有除錯資料，並記錄來源、適用環境與失效風險。

禁止使用正式環境資料、個人資料與脆弱的固定 ID。自動化測試也應先找合適資料，但仍優先自行建立可控資料，不依賴偶然存在的環境狀態。

## 手動驗證與唯讀 SQL

手動驗證需要資料時：

- 依實際 Model、Repository、migration 或 table schema 證據撰寫查詢，不猜測表名與欄位。
- 標示專案／倉庫、相對證據路徑、執行環境、參數、預期條件與讀取範圍。
- 預設只提供參數化 `SELECT`，只取驗證所需欄位，不顯示憑證、個資或其他敏感欄位。
- 測試設計不產出 `INSERT`、`UPDATE` 或 `DELETE`。若情境需要建立或修改資料，只記錄需由使用者依目標專案既有流程另行處理，以及必要的前置條件與清理要求。

無手動 SQL 或無某類驗證時，使用普通文字寫「無。依據：…」，不保留未勾選項目假裝尚未完成。

同樣規則適用於手動、權限、環境與上線觀察；這些類別與 SQL 全部不適用時，整個章節只保留一行依據，不展開空表格、空欄位或未完成清單。

## 實作後強制回填

初版文件必須保留「自動化測試實施結果」二級標題，但只填「尚未進入實作；實作後依情境 ID 回填」，不展開表格，也不虛構命令或結果。

在目標專案建立或修改自動化測試後，才以精簡表格取代該行並更新原測試設計：

1. 保留原始策略與情境，不用實作結果改寫歷史設計。
2. 逐一回填有自動化的情境 ID、狀態、實際測試檔案與名稱、實際命令與結果、設計偏差，以及剩餘手動／環境驗證與風險。
3. 實際資料策略與原設計相同時只填「依原設計」；只有偏離「測試資料策略」時才說明實際方式、原因與影響。
4. 經核准不自動化的情境，逐一記錄情境 ID、專案／倉庫、核准理由、替代驗證與執行位置；完全沒有時才寫「無。依據：…」。
5. 沒有這個章節，或有自動化卻尚未回填必要結果時，不得將測試交付標記為完成。
6. 後續需求或設計迭代建立新版文件，不覆蓋原文件。

## 交付測試清單

交付清單只用一張表列出情境 ID 與狀態摘要。初版只能標示「未實作」或「待執行」，不得虛構結果；實作後的檔案、名稱、命令與結果統一回填「自動化測試實施結果」。測試程式碼隨實作程式碼交付，清單不再次描述步驟、資料、限制或風險。

## 閱讀性與安全

- 預設使用繁體中文、短章節與可掃描表格。
- 預設先刪除重複內容，再考慮增加章節；小型需求不因範本欄位而膨脹。
- 步驟只寫可執行動作，預期結果只寫可觀察證據，避免長段背景與重複說明。
- 公開內容只使用虛構、中立資訊；不放入真實公司、專案、需求 ID、內部 URL、本機絕對路徑、憑證或個人資料。
