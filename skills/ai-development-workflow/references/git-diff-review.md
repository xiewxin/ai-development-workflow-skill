# Git Diff 審查指南

## 權限與範圍

Git Diff 審查預設唯讀。只有使用者明確要求「審查並修復」，或審查後確認修復，才修改程式碼。即使已有修復授權，涉及對外合同、擴大需求、破壞性操作或偏離已核准需求計畫時，仍須先確認。

審查期間不執行 `reset`、`clean`，也不覆寫使用者工作樹。會改動檔案的格式化、產碼或測試命令，需先確認影響或改用安全方式。

## 判定審查基準

基準優先級依序為：

1. 使用者明確指定。
2. MR／PR 的 target branch。
3. 可驗證的 upstream 或 default branch。
4. 無法可靠判定時，攜帶已查證據詢問使用者。

功能分支通常從 `master` 建立時，預設使用 `git diff origin/master...HEAD`，由 Git 透過 merge-base 比較共同祖先後的變更。若 MR／PR、upstream 或分支歷史證明實際基準是 `develop`、`release` 或其他分支，改用該實際基準，不固定套用 `master`。

先確認 remote ref 是否新鮮，可檢查 remote、tracking branch、remote ref 最新 commit 與 reflog 更新時間。只有在網路、權限及目前任務允許，且更新不會影響使用者工作樹時，才安全 fetch 對應 remote ref。無法更新時，揭露已知 commit、時間與限制，並將結論標示為基於該快照；不得聲稱已是最新版本。

## 分開盤點變更

不要把不同來源的變更混成一個 Diff。逐項檢查並在輸出列明範圍：

- 已提交變更：`git diff <base>...HEAD`
- 提交清單：`git log <base>..HEAD`
- staged：`git diff --cached`
- unstaged：`git diff`
- untracked 與範圍狀態：`git status --short --untracked-files=all`，再讀取本次範圍內檔案

先看檔案清單與統計，再讀完整相關 Diff、必要上下文及測試。若檔案同時出現在多個範圍，分別說明其 committed、staged 或 unstaged 狀態；不擅自合併、回復或清理。

## 對照需求與證據

有需求計畫或測試設計時，逐項對照需求與驗收條件、檔案級實作計畫、測試、文件，以及已核准偏差。需求計畫只代表意圖，仍以目前程式碼、Diff 與可重現測試驗證實作。

沒有需求計畫或測試設計時，依需求來源、根目錄規範、已由程式碼驗證的 `docs/agents/`、實際程式碼與測試推導預期行為。證據不足且會影響判定時，列入待確認問題，不以猜測補齊。

## 審查維度

- 功能正確性與實作完整性：主流程、邊界、錯誤路徑、狀態與資料流是否符合需求。
- 對外合同與相容性：API、RPC、事件、資料結構、設定與既有呼叫端是否維持約定。
- 安全：輸入、權限、敏感資料、注入、路徑與破壞性行為是否受控。
- 程式碼品質：命名、型別、例外、重複、可讀性與可維護性是否清楚一致。
- 複用性、設計規範與分層：是否沿用穩定能力，且職責放在正確層級。
- 簡潔優雅：是否存在多餘抽象、無職責轉發、重複分支或過度設計。
- 效能與可靠性：N+1、重複 RPC、併發、交易、快取、資源用量及失敗恢復是否合理。
- 測試與文件：關鍵行為、回歸範圍、測試穩定性、使用方式及合同文件是否同步。

「簡潔優雅」必須可驗證：更少重複查詢、分支或轉換；職責歸屬正確；複用穩定能力；抽象有實際使用場景；容易測試與修改；沒有無職責轉發層。不得把個人風格偏好當成問題。

## 嚴重度

- **P0**：可能造成資料損毀、安全事故或服務中斷，必須阻斷交付。
- **P1**：違反核心需求、主流程或外部合同，或會影響高風險正式環境。
- **P2**：在明確觸發條件下造成錯誤，或形成重要邊界、效能或維護問題。
- **P3**：低風險、局部且不阻斷交付的品質改進。

## Finding 合同

每個 P0－P2 使用固定短格式，通常不超過五行：

```markdown
### [P?][類型] 具體標題
- 位置：`file:line` 或最窄 symbol
- 影響：具體後果與支持判定的證據
- 觸發：可重現條件與必要證據
- 建議：修正方向，不貼大段程式碼
```

所有可行動缺陷，包括功能、測試缺口、文件缺口與實作計畫偏差，都依嚴重度只回報一次：P0－P2 使用上述固定格式，P3 集中於「改進建議」且每項一行。不得在後續區段換名、降級或重複列出。

只回報具備具體證據、影響、觸發條件與修正方向的問題。證據融入「影響」或「觸發」，不另寫長段。多個同源問題合併，不重複列出。

## 輸出順序

依序輸出：問題（P0 至 P2，由高到低）→ 待確認問題 → 覆蓋限制 → 背景偏差 → 改進建議（P3）→ 簡短總結與剩餘風險。沒有內容的類別可省略。

「待確認問題」只放缺少關鍵事實、尚不能定性為 finding 的問題；「覆蓋限制」只放未執行或無法取得、但尚無證據顯示缺陷的驗證範圍；「背景偏差」只放歷史計畫或文件差異中，尚不能判定為實作缺陷的背景資訊。三者不得重複 finding 或彼此重複；一旦已有足夠證據形成可行動缺陷，就移入對應 P0－P3 並只回報一次。

沒有問題時，明確寫「未發現需要修正的明確問題」，並簡短列出審查範圍、需求符合、設計／複用、測試、文件、具證據的適度鼓勵與剩餘風險。這表示目前證據未顯示明確缺陷，不保證零缺陷，也不為填滿格式硬找問題。

## 修復後驗證

取得修復授權後，只處理已確認範圍。修復完成需重跑相關測試、重新盤點 committed／staged／unstaged／untracked 範圍，並對完整 Diff 再審查一次。輸出保持短：finding 通常五行以內，總結只保留決策、驗證結果與剩餘風險，不使用空泛稱讚。
