# Git Diff 審查指南

## 權限與範圍

Git Diff 審查預設唯讀。只有使用者明確要求「審查並修復」，或審查後確認修復，才修改程式碼。即使已有修復授權，涉及對外合同、擴大需求、破壞性操作或偏離已核准需求計畫時，仍須先確認。

審查期間不執行 `reset`、`clean`，也不覆寫使用者工作樹。會改動檔案的格式化、產碼或測試命令，需先確認影響或改用安全方式。

## 判定審查基準

基準優先級依序為：

1. 使用者明確指定。
2. MR／PR 的 target branch。
3. 可驗證的 upstream 或 default branch。
4. 無法可靠判定時，攜帶已查證據詢問使用者。

功能分支通常從 `master` 建立時，預設使用 `git diff origin/master...HEAD`，由 Git 透過 merge-base 比較共同祖先後的變更。若 MR／PR、upstream 或分支歷史證明實際基準是 `develop`、`release` 或其他分支，改用該實際基準，不固定套用 `master`。

先確認 remote ref 是否新鮮，可檢查 remote、tracking branch、remote ref 最新 commit 與 reflog 更新時間。只有在網路、權限及目前任務允許，且更新不會影響使用者工作樹時，才安全 fetch 對應 remote ref。無法更新時，揭露已知 commit、時間與限制，並將結論標示為基於該快照；不得聲稱已是最新版本。

## 分開盤點變更

不要把不同來源的變更混成一個 Diff。逐項檢查並在輸出列明範圍：

- 已提交變更：`git diff <base>...HEAD`
- 提交清單：`git log <base>..HEAD`
- staged：`git diff --cached`
- unstaged：`git diff`
- untracked 與範圍狀態：`git status --short --untracked-files=all`，再讀取本次範圍內檔案

先看檔案清單與統計，再讀完整相關 Diff、必要上下文及測試。若檔案同時出現在多個範圍，分別說明其 committed、staged 或 unstaged 狀態；不擅自合併、回復或清理。

## 對照需求與證據

有需求計畫或測試設計時，逐項對照需求與驗收條件、檔案級實作計畫、測試、文件，以及已核准偏差。需求計畫只代表意圖，仍以目前程式碼、Diff 與可重現測試驗證實作。

對完整 Diff 做範圍對帳：每個變更檔案與行為群都標記為「已核准計畫」「已核准偏差」或「未映射」。同一檔案同時包含核准與未核准行為時分開判定；測試通過、提交訊息相同或文件已標記完成，都不能取代需求來源與核准證據。未映射變更依影響形成 finding，不能藏在覆蓋限制或背景偏差。

若變更共用格式化、正規化、權限、設定或其他高擴散規則，核對實際存在的正常入口、舊快取／歷史資料、外部服務／RPC 及 fallback、legacy／管理端／內部／demo、通知與背景任務。若候選值替換為不同語義或形狀的正式值，例如 ID 變成 URL，同時核對欄位名、consumer 行為、schema／型別、fixture 與文件；不能以非空測試代替合同確認。

沒有需求計畫或測試設計時，依需求來源、根目錄規範、已由程式碼驗證的 `docs/agents/`、實際程式碼與測試推導預期行為。證據不足且會影響判定時，列入待確認問題，不以猜測補齊。

## 審查維度

- 功能正確性與實作完整性：主流程、邊界、錯誤路徑、狀態與資料流是否符合需求。
- 對外合同與相容性：API、RPC、事件、資料結構、設定與既有呼叫端是否維持約定。
- 安全：輸入、權限、敏感資料、注入、路徑與破壞性行為是否受控。
- 程式碼品質：命名、型別、例外、重複、可讀性與可維護性是否清楚一致。
- 複用性、設計規範與分層：是否沿用穩定能力，且職責放在正確層級。
- 簡潔優雅：是否存在多餘抽象、無職責轉發、重複分支或過度設計。
- 效能與可靠性：N+1、重複 RPC、併發、交易、快取、資源用量及失敗恢復是否合理。
- 測試與文件：關鍵行為、回歸範圍、測試穩定性、使用方式及合同文件是否同步。

「簡潔優雅」必須可驗證：更少重複查詢、分支或轉換；職責歸屬正確；複用穩定能力；抽象有實際使用場景；容易測試與修改；沒有無職責轉發層。不得把個人風格偏好當成問題。

## 嚴重度

- **高**：可能造成資料或安全事故、服務不可用、核心需求失敗或外部合同破壞，必須阻斷交付。
- **中**：在明確條件下造成錯誤、重要回歸、效能或維護風險，原則上應在交付前修正或取得風險接受。
- **低**：局部、低風險且不阻斷交付的品質或可維護性改善。
- **緊急標記**：只有處理時效明顯短於一般排程時附加 `緊急`；緊急程度不改變問題嚴重度。

## 多來源審查

Superpowers、Spec Kit、OpenSpec、BMAD、其他 Provider reviewer、人工 review 與本 Skill 審查都只是原始證據來源，不各自維護最終問題清單。協調代理必須：

1. 以目前程式碼、完整 Diff、需求、測試與文件驗證每項原始結論。
2. 排除無證據意見，並依根因、影響、處置、驗證與關閉邊界去重。
3. 將可行動問題只分配到一組 `REV-*` 清單，並在後續複審只更新該清單的狀態。

同根因、同處置且可由同一次驗證關閉時合併；跨倉庫、跨合同或需獨立修正／驗證／接受風險時分開編號並以關聯 `REV-*` 連結。Provider 自身的嚴重度、ID 或通過結論不直接沿用；以本指南的證據、嚴重度與狀態合同正規化。

## Finding 合同

每個可行動問題都取得目前對話內唯一且穩定的 `REV-*` 編號。編號依首次確認順序由 `REV-001` 遞增；同一問題複審時沿用原編號，不因嚴重度、狀態或涉及倉庫改變而重新編號。本合同只保證目前對話、目前需求及其涉及的多個倉庫內穩定，不建立跨對話、跨需求的永久議題編號。

使用固定短格式，通常不超過六行；需要緊急標記時放在嚴重度之後：

```markdown
### [REV-001][高][類型] 具體標題
- 狀態：待處理
- 位置：`file:line` 或最窄 symbol
- 影響：具體後果與支持判定的證據
- 觸發：可重現條件與必要證據
- 建議：修正方向，不貼大段程式碼
```

所有可行動缺陷，包括功能、測試缺口、文件缺口與實作計畫偏差，都只回報一次並使用上述格式。不得在後續區段換名、降級或重複列出。

只回報具備具體證據、影響、觸發條件與修正方向的問題。證據融入「影響」或「觸發」，不另寫長段。同一根因、同一修正方式且能由同一次驗證關閉的問題合併；跨倉庫、跨合同或需獨立修正／驗證／接受風險時，分配不同編號並以「關聯：`REV-*`」互相連結。待確認問題尚未具備可行動證據時不預占編號。

## 問題狀態

- **待處理**：問題已確認，尚未完成修正或決策。
- **已修正待驗證**：已有對應變更，但尚未取得可重現的最新驗證證據。
- **已關閉**：修正已通過相關測試與完整 Diff 複審，或新證據證明原問題不成立。
- **接受風險**：使用者已明確理解影響並決定不修正；記錄決策依據與剩餘風險。

狀態只能依新證據前進或重新開啟；修改程式碼本身只能進入「已修正待驗證」，不能直接關閉。嚴重度只有在影響或觸發證據改變時調整，仍沿用原編號並簡短說明原因。

## 複審輸出

首次審查輸出全部 finding。後續複審先列一行狀態摘要，再只展開狀態改變的既有 finding、新 finding、重新開啟項目及仍需決策的接受風險；未變更的已關閉項目不重貼全文。使用者直接引用 `REV-*` 時，針對該問題及其明示關聯項驗證，不要求重新描述問題。

複審仍需重新盤點完整 committed、staged、unstaged 與 untracked 範圍，避免定向修正引入新問題。沒有新問題且指定 finding 已關閉時，輸出驗證證據、狀態與剩餘風險即可。

## 輸出順序

依序輸出：問題（高至低）→ 待確認問題 → 覆蓋限制 → 背景偏差 → 簡短總結與剩餘風險。沒有內容的類別可省略。

「待確認問題」只放缺少關鍵事實、尚不能定性為 finding 的問題；「覆蓋限制」只放未執行或無法取得、但尚無證據顯示缺陷的驗證範圍；「背景偏差」只放歷史計畫或文件差異中，尚不能判定為實作缺陷的背景資訊。三者不得重複 finding 或彼此重複；一旦已有足夠證據形成可行動缺陷，就移入對應嚴重度並只回報一次。

沒有問題時，明確寫「未發現需要修正的明確問題」，並簡短列出審查範圍、需求符合、設計／複用、測試、文件、具證據的適度鼓勵與剩餘風險。這表示目前證據未顯示明確缺陷，不保證零缺陷，也不為填滿格式硬找問題。

## 修復後驗證

取得修復授權後，只處理已確認範圍。修復完成需重跑相關測試、重新盤點 committed／staged／unstaged／untracked 範圍，並對完整 Diff 再審查一次。輸出保持短：finding 通常六行以內，總結只保留決策、驗證結果與剩餘風險，不使用空泛稱讚。
