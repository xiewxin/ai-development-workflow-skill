# AI Development Workflow Skill 設計規格

## 目標

建立一個可公開安裝、同時支援 Codex 與 Claude Code 的 `ai-development-workflow` Skill，協助使用者完成需求計畫、測試設計與 Git Diff 審查，並在完整流程中協調目前代理依核准計畫實作。Skill 本身不提供獨立程式碼產生器，而是約束目前代理使用目標專案既有工具、規範與驗證方式完成實作。

## 核心原則

- 預設使用繁體中文產出文件、註解、測試名稱、日誌及內部錯誤訊息；目前使用者的明確語言要求優先，外部固定合同與識別字保持原樣。若目標專案明定其他語言且與目前要求衝突，先指出規範與受影響範圍並取得確認，不靜默混用語言。
- 先讀需求與專案證據，再提出結論；無法驗證的內容明確標記為假設或待確認。
- 先複用、後新增；需求計畫必須包含影響分析、複用分析及風險識別。
- 計畫、測試設計及審查結果保持簡潔、可執行，避免重複描述與長篇輸出。
- 預設不覆蓋歷史計畫；新迭代建立新文件並引用舊計畫。
- 公開 Skill 不含憑證、個資、內部連結、真實專案資訊或真實業務資料。

`SKILL.md` 只保留共用流程、模式選擇、確認點與 reference 導航。細節按模式載入，避免一次占用過多上下文。`agents/openai.yaml` 僅提供 Codex UI metadata，不改變核心 Skill 行為。

## 輸入與證據優先級

需求來源預設為任務管理系統，也接受原型、文件、表格或對話說明。證據優先級如下：

1. 目前對話中的明確說明。
2. 需求與驗收標準。
3. 原型、知識庫及表格。
4. 目前程式碼、測試及已驗證文件。
5. 歷史計畫，只用於理解設計初衷。

預設以目前工作目錄為主倉庫。先分析需求、專案規範、相關模組文件及目前程式碼；只有發現目前專案無法閉環的具體證據時，才詢問是否補充其他專案。詢問時列出缺口與證據，未提供其他專案也繼續完成已知範圍並標記待驗證項目。

## 執行模式

| 模式 | 選擇／升級條件 | 主要輸出 | 寫入規則 | 完成條件 |
|---|---|---|---|---|
| 需求計畫 | 明確要求計畫、方案或影響分析 | `<topic>.md` | 使用者明確要求建立計畫時，寫入前告知路徑即可；若由代理推斷需要落盤，先取得確認 | 計畫已建立且待確認事項已標記 |
| 測試設計 | 明確要求測試設計、測試案例或交付測試 | `<topic>-test-design.md` | 使用者明確要求建立測試設計，或已核准需求計畫要求建立時，寫入前告知路徑即可；其他情況先取得確認 | 測試情境、資料策略與驗證映射完整 |
| Git Diff 審查 | 明確要求審查目前分支、差異或 MR/PR | 對話中的短格式審查結果 | 預設唯讀，不寫入檔案、不修改程式碼 | 審查範圍、發現及剩餘風險已說明 |
| 完整流程 | 明確要求端到端完成／實作需求，或明確指定完整流程 | 計畫、測試設計、經核准實作、審查結果 | 文件按上述規則寫入；程式碼實作必須在計畫獲得明確核准後開始 | 實作、驗證、文件回填及審查完成，或明確標記未完成／阻塞項 |

文件建立後的狀態更新屬於原工作流的生命週期，不需要每次另行確認。任何新的程式碼修改、未核准的方案偏離、破壞性操作或遠端操作仍需遵循目標專案規範及使用者授權。

模式路由規則：

1. 使用者明確指定一個或多個模式時，只執行指定模式，依需求計畫、測試設計、Git Diff 審查的自然順序處理，不自動升級成完整流程。
2. 「實作／完成這個需求」視為端到端完整流程；如果已有獲核准計畫，從相應未完成階段繼續，否則先建立計畫並等待核准。
3. 「審查並修復」仍屬 Git Diff 審查模式，該措辭已包含對已確認且範圍內問題的修復授權，不需要再次詢問；若修復會改變外部合同、擴大需求、執行破壞性操作或偏離既有計畫，仍須停下確認。
4. 只要求計畫加測試設計時，不進入程式碼實作。
5. 無法可靠判斷模式時只詢問一個範圍問題，不擅自選擇完整流程。

### 需求計畫

- 先讀專案根規範文件、相關實作說明、程式碼與測試；歷史計畫僅作參考。
- 必填章節：來源與目標、現況、影響分析、複用分析、方案與決策、風險、檔案級計畫、實作清單、測試與驗證、文件改動、待確認事項。
- 依專案既有慣例寫入 `docs/plans`；沒有慣例時使用 `docs/plans/YYYY-MM-DD-<topic>.md`。
- `<topic>` 優先沿用專案既有命名；沒有慣例時，以需求主題產生簡短、穩定的 kebab-case slug。若目標檔案已存在，不覆蓋舊檔，依序使用 `-v2`、`-v3`，並在新計畫引用前一版。
- 寫入前說明路徑；實作前必須取得使用者核准。
- 計畫狀態使用 `草擬`、`已核准`、`實作中`、`已完成`、`部分完成`、`已阻塞` 或 `已取代`。完成是指核准範圍已實作、相關驗證已執行且文件已回填；若仍有必要工作，使用相應的非完成狀態。
- 完成後在末尾加入簡短的「AI 協作成效」。不維護個人或跨需求統計。

### 測試設計

- 預設建立獨立的 `docs/plans/YYYY-MM-DD-<topic>-test-design.md`；同名檔案已存在時沿用需求計畫的 `-v2`、`-v3` 防覆蓋規則。
- 每個情境記錄前置條件、步驟、預期結果、驗證類型及對應測試路徑。
- 自動化測試優先使用 mock、factory、fixture 或測試自行建立的資料；避免真實資料及脆弱固定 ID。
- 手動驗證需要資料時，可提供唯讀、參數化且不含敏感欄位的查詢範例。
- `/tests` 指目標專案實際採用的自動化測試目錄，不代表固定框架或命令。實作階段依目標專案慣例建立或更新自動化測試；完成後追加實際測試名稱、資料策略、指令、結果、偏差及剩餘風險，不覆蓋原始設計。

### Git Diff 審查

- 基準優先級：使用者指定、MR/PR 目標、可驗證的預設分支，否則帶證據詢問。
- 功能分支預設使用 `<base>...HEAD` 審查共同祖先後的已提交變更，另行檢查 staged 與 unstaged 內容。
- 審查功能正確性、計畫符合度、程式碼品質、複用性、設計規範、簡潔性、效能可靠性、測試及文件。
- 優先級定義：P0 為資料損毀、安全事故或服務中斷阻斷項；P1 為核心需求、主要流程、對外合同或高風險正式環境問題；P2 為有明確觸發條件的功能、重要邊界、效能或可維護性問題；P3 為低風險、局部品質改進。
- 每個問題使用短格式：`[優先級][類型] 標題`，以及位置、影響、觸發、建議。
- P0 至 P2 逐項輸出；P3 集中在「改進建議」。只回報具有證據、影響及修正方向的問題。
- 沒有問題時說明審查範圍、通過依據、適度總結與剩餘風險，不硬找問題。
- 預設唯讀；使用者確認修復後才修改程式碼、測試或文件。

### 完整流程

依序協調需求計畫、測試設計、經核准的實作及 Git Diff 審查。文件按模式矩陣處理，程式碼實作以計畫核准為必要確認點。發現新事實或需要偏離已核准方案時，先列出證據、影響與新選項，取得確認並更新計畫後再繼續。

## 分層與文件規則

分層首先遵循目標專案規範與已驗證的穩定架構。只有當目標專案本身採用 Controller、Logic、Service、Repository 概念且沒有相反規範時，才以 `Controller -> Logic -> Service -> Repository` 及「Repository 原則上只由 Service 呼叫」作為候選預設；其他技術棧不得套用此結構。不為形式統一而做無關重構。

需求在哪個專案實作，就依該專案既有文件慣例判斷是否更新實作說明；穩定行為已改變且已有相關文件時必須同步。新模組但專案沒有相應文件慣例時，在需求計畫提出文件路徑與必要性，取得確認後再建立。只有呼叫關係的專案只在需要文件同步時記錄合同、呼叫方向與失敗處理。跨專案需求預設只在主倉庫建立需求計畫與測試設計。

## 安全與隱私

公開套件與執行期產物使用不同邊界：

- 公開 GitHub 套件、範例、測試 fixture 及開發文件只使用完全虛構、中立的名稱及資料，不得含真實專案名稱、需求 ID、內部連結、內部路徑或真實業務資料。
- Skill 在使用者的目標倉庫執行時，可以在該倉庫的私有計畫、測試設計與審查結果中記錄完成任務所需的專案名稱、相對程式碼路徑、需求來源及合同證據；只保留必要資訊，不將其複製回公開 Skill 倉庫。
- 無論公開或私有產物，都不得複述或寫入 API Key、Token、Cookie、密碼、私鑰、個資或不必要的真實資料。發現時使用占位符，說明來源位置與風險但不回顯敏感值。
- 發現公開套件含疑似敏感資訊時停止發布並回報精確檔案及規則，不自動重寫 Git 歷史。
- 建立或推送遠端前執行發布檢查；遠端建立與推送均需要使用者明確確認。

## 驗證策略

- Skill Creator 的外部 `quick_validate.py`：Codex 開發環境可用時，從已安裝的 `skill-creator` 取得並驗證 Skill 名稱、frontmatter 及基本格式；它不是公開套件的執行期依賴。不可用時由本專案發布檢查覆蓋必要結構與 frontmatter，並在結果標記未執行外部驗證器。
- `validate-publication.sh`：靜態檢查必要目錄、`SKILL.md` frontmatter、失效的相對連結、未完成占位符、繁簡體禁用詞及疑似憑證、個資、內部 URL；只輸出檔案與規則，不回顯疑似 secret。繁體檢查使用可讀的常見簡體詞彙清單，外部固定合同可透過本機 allowlist 排除並記錄理由；敏感資訊規則不可用 allowlist 略過。
- `test-validate-publication.sh`：使用暫存虛構檔案驗證掃描器能阻擋與放行預期內容，不接觸真實資料。
- 代理情境：先在無 Skill 時建立基準，再使用相同的虛構輸入載入 Skill 重跑。每個情境使用明確檢核表人工判定，不以文字完全一致作為標準。

代理情境的最低通過條件：

1. 需求計畫包含必填章節、證據路徑、複用判斷及風險，且只在有具體跨專案缺口時詢問補充範圍。
2. 測試設計將情境映射到自動化、手動或環境驗證，說明測試資料策略，並保留實作後回填位置。
3. Git Diff 審查能指出預植的功能或設計問題，使用短格式且不輸出無證據的風格偏好。
4. 遇到憑證、個資或內部資訊時停止公開輸出並以占位符替代，不複述敏感值。
5. 無 Skill 基準用於找出可重現的缺口或不一致，不為製造失敗而扭曲情境。若所有基準條件原本已通過，先擴充合理變體或重新評估 Skill 的必要性；載入 Skill 後不得退化，且已發現的缺口必須改善。若無法改善，修訂 Skill 後重測。

Claude Code 使用相同 `SKILL.md` 開放格式；本機沒有 Claude Code 執行環境時，明確標記尚未實機驗證的剩餘風險。

## 非目標

- 不維護個人 CSV、團隊儀表板或跨需求統計。
- 不綁定特定公司、專案、程式語言或任務管理工具。
- 不自動建立、推送或修改 GitHub 遠端。
- 不以 Skill 取代目標專案自身的規範與測試工具。
